What This Is

A network monitoring system built in Python that runs in the background, pings every device on your home network, and shows everything on a live dashboard. It tracks which devices are online, how fast they are responding, what ports they have open, and alerts you when something goes down or gets slow.

Why I Built It

Most home networks have no monitoring at all. You find out something is broken when someone tries to use it and it does not work. I wanted something that would catch problems on its own without needing to check manually.

There is also a difference between a device being completely offline versus just getting slow. A device that starts responding slower and slower is usually a sign something is wrong before it fully goes down. Just knowing up or down is not enough — you need the latency over time too.

On top of that I realized I did not actually know everything that was on my network or what ports those devices had open. Some of that is a security concern you would not notice otherwise, like a device running Telnet or having SMB exposed.

Problems I Ran Into & their Solutions

Four real bugs that came up during development, and how each one got fixed.

The ARP table was hanging on macOS

The first version used arp -a to get MAC addresses. On macOS that command tries to do a DNS lookup on every single entry before it returns anything, so it would sit there for several seconds and sometimes come back with incomplete results. Switching to arp -an tells it to skip DNS and just return the raw table. That fixed it immediately.

IP addresses were showing up where MAC addresses should be

Some devices respond to a ping but are not in the ARP table yet. This happens with devices that clear their ARP entries quickly. When that happens there is no MAC to store, so the monitor falls back to using the IP address as the device identifier and stores it in the MAC column. That caused IP addresses to show up in the MAC column on the dashboard. Every place in the code that reads that column had to be updated to detect whether the value is actually a MAC or an IP and handle it differently.

Device names were leaking through in public mode

The dashboard has a public mode that hides personal info like full IP addresses and MAC addresses. That worked fine on the device table but the latency chart and the availability heatmap were reading labels straight from the raw database. So if a device was labeled something like "Mac mini" in the config, that name would still show up on the chart even with public mode turned on. The fix was to swap all labels out for generic vendor categories like "Apple device" or "Router" before the chart data even gets built, instead of trying to find and remove sensitive patterns after the fact.

Technical Debt: Data Migration for Public View

The project originally ran on a local SQLite database. That worked fine for personal use, but once I wanted to host the dashboard publicly on Streamlit, the database itself would have had to be published alongside the code — which meant exposing all the collected device data, not just hiding it with public mode. Because the original choice didn't account for public hosting, fixing it wasn't a quick patch — it meant migrating the database to Neon (hosted PostgreSQL) and rewriting every piece of code that read or wrote to SQLite. The collector now writes directly to Neon instead of a local file, and the Streamlit deployment connects to it using a Neon passkey connection string stored in the app's advanced settings, so the actual data never has to live in the repo.

The chart got slow after a few days of data

The latency chart was loading the entire ping history from the database on every page refresh. After a few days of logging that started taking noticeably long. The fix was a separate database query that only pulls the last 24 hours for the chart, while the full history still gets loaded separately for uptime percentages and alerts where you need all the data.
